# 设计规格：主窗口尺寸、标题栏与独立设置窗口

**日期**：2026-05-08  
**状态**：已与用户对齐（主方案按前期 brainstorm 推荐；设置复杂度为 B）  
**范围**：Flutter 桌面应用（macOS 为主；Windows 行为在实现阶段对齐）

---

## 1. 目标

1. **去掉系统标题栏的视觉占位**：主窗口内容可延伸至窗口顶部；交通灯与拖拽行为有明确、可操作的方案（见 3）。
2. **主窗口紧凑包裹中央番茄卡片**：初始尺寸贴近内容；卡片悬停展开时，**窗口客户区同步增大**，避免裁切；更新策略避免每帧抖动（见 4）。
3. **设置与统计等重界面（复杂度 B）**：使用 **独立设置窗口**，主窗口保持「计时卡片」体量，不把多 Tab、图表、长列表塞进同一小窗。

---

## 2. 非目标

- 不在本规格内定义设置页具体信息架构与视觉稿（仅约束「独立窗口 + 足够布局空间」）。
- 不强制本阶段完成 Linux 端与 macOS 100% 行为一致（可在实现计划中列为后续项）。

---

## 3. 标题栏（系统 chrome）

**术语**：用户所指「红框区域」为 **系统标题栏（title bar）** 与 **交通灯（window controls）** 所在区域。

**方向**：

- **推荐**：**透明标题栏 + 全尺寸内容区（full-size content view）**，必要时 **标题文字隐藏**，使 Flutter 绘制的背景与圆角卡片视觉上连续。
- **交通灯**：二选一，在实现计划中择一并写清验收标准：
  - **A**：保留系统交通灯，通过 `fullSizeContentView` 让按钮叠在 Flutter 内容之上（需在 Flutter 顶部为交通灯预留 **安全内边距**，避免与卡片内「关闭」等控件重叠）。
  - **B**：无边框或极简 chrome，交通灯由 **Flutter 插件** 绘制在内容区内，并补充 **窗口拖拽区域**（如顶部 DragRegion）。

**验收**：主窗顶部无单独一条与内容割裂的灰色标题条（除非刻意设计的品牌条）；窗口可关闭、最小化、可拖拽移动。

---

## 4. 主窗口尺寸与卡片展开

**原则**：窗口 **content 尺寸** 由「当前主界面所需矩形 + 统一外边距」推导；设置 **最小/最大** 宽高，避免异常放大。

**与悬停动画的配合**：

- 不在每一 layout frame 无节流地调用原生 `setContentSize`（防抖动与耗电）。
- **推荐策略**：动画期间可用 **略大于收起态的预留** 或 **短间隔节流（如 50–100ms）** 合并多次尺寸请求；**动画结束** 再 **精确对齐** 一次最终尺寸。
- **裁切**：以「展开态卡片 + 外边距」为下限验收，hover 展开后 1px 不被窗口裁切。

---

## 5. 设置：独立窗口（复杂度 B）

**决策**：**独立设置窗口**（或平台惯用的 **Sheet** 仅当与「独立窗」实现成本接近时再评估；默认 **独立 NSWindow / 第二 Flutter 引擎或单引擎双窗** 以工程可行为准）。

**理由简述**：B 类含统计图表、任务列表、多 Tab，与「主窗小而专注」冲突；独立窗符合系统「偏好设置」心智，主计时窗可保持角落挂件形态。

**状态同步**：

- 主题、语言、计时相关配置等 **单一数据源**（如已存在的 state / repository）；设置窗关闭或即时写回，避免两窗逻辑分叉。
- 明确 **未保存更改** 行为（保存即写、或显式「应用/保存」按钮），在实现计划中落一条规则即可。

**尺寸**：设置窗使用 **独立默认尺寸与最小尺寸**（大于主窗），不强行与主窗同宽；用户可手动拉大。

---

## 6. 平台与实现提示（非绑定）

- **macOS**：`MainFlutterWindow.swift` 中配置 `titlebarAppearsTransparent`、`styleMask`、必要时 `titleVisibility`；尺寸通过已有或新增的 **platform channel** / 社区 `window_manager` 等与 Flutter 联动。
- **Windows**：`win32_window` 侧对齐无边框或自定义标题栏与 **DWM 拖拽** 区域，避免与 macOS 行为长期分叉过大。

---

## 7. 风险与依赖

- **双窗 Flutter**：需确认当前工程是单 `FlutterViewController` 还是可挂载第二窗口；若成本高，可采用 **单引擎 + 原生第二窗口内嵌同一引擎** 或 **设置路由在独立 `Navigator` 的窗口** 等方案，在 **implementation plan** 阶段选定一种并记录取舍。
- **无障碍与焦点**：设置窗打开时，焦点与屏幕阅读器顺序需可接受（实现阶段自测）。

---

## 8. 验收清单（摘要）

- [ ] 主窗顶部无割裂的系统灰条；可拖拽、可关闭。
- [ ] 默认主窗明显小于当前「大块留白」状态，且包住番茄卡片。
- [ ] 悬停卡片展开后无裁切；动画过程无明显窗口尺寸抖动。
- [ ] 从主窗打开设置 → **独立窗口**；可容纳多 Tab、列表与图表布局。
- [ ] 设置与主窗配置一致、无长期分叉的两套默认值。
